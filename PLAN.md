## Decisions resolved

- **Q-A (README tone):** terse-and-internal.
- **Q-B (License):** MIT.
- **Q-C (Branch model):** framework ships BOTH options
  (single-branch default + three-branch model); init script
  asks which to wire.
- **Q-D (CHANGELOG discipline):** both — Conventional Commits
  for commit subjects, Keep-a-Changelog for `[Unreleased]`
  blocks and release notes.
- **Q-E (Skill dedupe default):** keep upstream skill unless
  framework-bundled version is newer (per `SKILLS.md` manifest).
  User can flip with `--prefer=framework|upstream|none`.

## Slice 1 (tracer bullet — fully detailed)

**Goal:** Ship the skeleton that proves the framework's shape.
A target repo can adopt the framework's `core/`, one addendum,
and the init script — and have a working bootstrap with no
follow-up required for the basic case.

### Files

- `README.md` — what it is, install steps, philosophy
- `LICENSE` — MIT
- `CHANGELOG.md` — `[Unreleased]` block, v0.1.0 scaffold entry
- `core/README.md` — index of core docs, tier-0/tier-1/tier-2
- `core/session-protocol.md` — full agent session rules, de-DixieData-fied
- `core/commit-and-branch.md` — commit shape + branch policy
- `core/laws.md` — non-negotiable laws (just the law statements,
  no stack history)
- `core/docs-index-scheme.md` — the 3-tier pattern as a reusable
  template
- `addenda/README.md` — when to load which addendum
- `addenda/go-htmx.md` — Go + HTMX + templ + chi + Wails patterns
  + dialog-guard helper templates + routebuilder/htmxattr/uiids
  patterns + bug catalog §1-§6 from DixieData COMMON_BUGS
- `issues/README.md` + `issues/label-taxonomy.md` +
  `issues/bug-template.md` + `issues/feature-template.md` +
  `issues/triage-workflow.md`
- `scripts/init.sh` — bash bootstrap, interactive question flow,
  idempotent
- `scripts/sync-labels.sh` + `scripts/backfill-labels.sh`
- `templates/` — boilerplate files for init.sh to copy
- `skills/SKILLS.md` — manifest shape + dedupe contract
- `skills/dedupe-skills.sh` (placeholder script + interface spec)

### Success criteria

- A user can `git clone agent-stack && bash scripts/init.sh /path/to/target`
  and answer 5 questions (stack, branch model, include issues scaffolding,
  include which addendum, which skills to bundle) and end up with a
  target repo that has `docs/agents/` + `core/` addendum + boilerplate
  AGENTS.md/CONTEXT.md + optional `.github/ISSUE_TEMPLATE/` + optional
  bundled skills.
- The init script is idempotent — re-running it on a bootstrapped
  repo only adds what's missing, never overwrites user edits.
- `FRAMEWORK_BOOTSTRAP.md` lands in the target repo with a checkbox
  list of every doc the team must read at session start.
- The Go/HTMX addendum loads cleanly against DixieData — every
  bug pattern it references maps to a specific section in
  DixieData's `docs/COMMON_BUGS.md`.
- `bash scripts/dedupe-skills.sh --dry-run` reports skill overlaps
  in a parseable table.

### Regression net

- `bash scripts/init.sh /tmp/test-target` against a scratch
  directory — succeeds, creates expected files.
- `bash scripts/init.sh /tmp/test-target` AGAIN — idempotent,
  no duplicates.
- `bash scripts/dedupe-skills.sh --dry-run` exits 0, prints
  table.
- Manual: read `core/session-protocol.md` and confirm zero
  DixieData-specific references (no "DixieData", no "Person
  Record", no "wailsapp/wails#2807" unless inside the
  Go/HTMX addendum).

## Subsequent slices (stub only)

- **Slice 2: Bundle skills.** Vendor the curated skill subset
  (tdd, rpci, diagnose, find-skills, tracer-bullets, grilling,
  bulk-read, tdd, systematic-debugging, learn-from-mistakes) under
  `skills/`. Each skill's SKILL.md is a near-copy of the upstream
  version with the agent-stack attribution header. `SKILLS.md`
  manifest pins source + version + checksum. Tested by
  `scripts/dedupe-skills.sh` exit 0 against a fixture with
  overlapping upstream skills.
- **Slice 3: Bootstrap scripts hardening.** `scripts/init.sh`
  gains dry-run mode, JSON output for CI, and a `--uninit` flag
  to reverse a bootstrap. `scripts/sync-labels.sh` is ported and
  tested against a scratch GitHub repo (via `gh repo create --template`).
  `scripts/backfill-labels.sh` is ported and tested.
- **Slice 4: Additional addenda.** `addenda/react.md`,
  `addenda/python.md`, `addenda/rust.md` — each one is a port
  of the Go/HTMX addendum with stack-specific bug patterns.
  Stub one (`addenda/react.md`) to prove the addendum shape
  generalizes; leave the others as follows.
- **Slice 5: Template repo + showcase.** Push agent-stack to
  GitHub as a public template. Ship a `examples/dixiedata-bootstrap/`
  subdirectory showing what a bootstrap looks like after running
  `init.sh` against DixieData (sanitized — no internal references).
- **Slice 6: Docs site.** MkDocs or Docusaurus config under `site/`
  for a browsable version of the docs. The framework is meant to
  be read; a site makes that easier.

## Anti-patterns the plan avoids

- **One-mega-slice shipping everything.** Slice 1 is the
  skeleton + addendum + issues scaffolding + scripts. Skills,
  other addenda, and the showcase come later. Tracer-bullet
  rhythm: prove the shape works before vending 15 skills.
- **Copying DixieData content verbatim.** Slice 1 explicitly
  tests that `core/` docs contain zero DixieData-specific
  references. The addendum is the only place stack-specific
  patterns live.
- **Baking the DixieData branch model into the core.** Slice 1
  carries both the single-branch and three-branch models in
  `commit-and-branch.md`; init script asks which to wire.
- **Shipping skills without a dedupe story.** SKILLS.md manifest
  ships in Slice 1 with the contract; skill bodies ship in Slice 2
  so the contract is testable independently.
- **Treating "issue framework" as separate from the rest.** It is
  wired into init.sh from Slice 1 (6th question); not bolted on
  later.
