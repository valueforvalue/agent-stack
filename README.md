# agent-stack

A drop-in framework of agent conventions, code-change contracts,
bug patterns, and tooling for AI-coding-assisted repos. Generalised
from real-world use on a Go + HTMX + templ + Wails app, applicable
to any language or stack via addenda.

## What you get

- **`core/`** — stack-agnostic rules: session protocol, commit &
  branch policy, RPCI flow, TDD discipline, feature protocol,
  cross-layer contract, laws, docs-index scheme, glossary
  discipline, bug-pattern catalog with per-layer stubs.
- **`addenda/`** — per-stack extensions. v1 ships
  `addenda/go-htmx.md`. Future addenda for React, Python, Rust
  drop in alongside.
- **`issues/`** — GitHub issue framework: 6-axis label taxonomy,
  bug + feature templates, triage workflow.
- **`skills/`** — bundled agent skills (curated subset). Each
  carries a SKILLS.md manifest entry with provenance + version +
  checksum. `scripts/dedupe-skills.sh` reports overlap with your
  installed skill catalog so you can prune on a case-by-case
  basis.
- **`scripts/init.sh`** — bootstraps a target repo with the
  framework. Asks 6 questions, copies the right files, generates
  `FRAMEWORK_BOOTSTRAP.md`. Idempotent.
- **`templates/`** — boilerplate files (AGENTS.md, CONTEXT.md,
  CHANGELOG.md, CONTRIBUTING.md, GitHub issue + PR templates).

## Install

```bash
git clone https://github.com/<your-org>/agent-stack.git
cd agent-stack
bash scripts/init.sh /path/to/your/repo
```

The init script is interactive. It asks:

1. Stack (drives which addendum is included)
2. Branch model (single-branch default vs three-branch)
3. Include issues scaffolding (GitHub labels + templates)
4. Which addendum(s) to include
5. Which bundled skills to include
6. README tone (terse vs polished)

Re-running it on an already-bootstrapped repo is safe — it
only adds what's missing.

## Why

Most "AI coding agent" docs are either too generic to be
useful ("always write tests") or too project-specific to be
portable ("see internal/handlers/foo.go"). agent-stack is
the middle layer: enough specificity to be load-bearing,
enough abstraction to drop into a fresh repo without
rewriting.

The procedural backbone (RPCI, TDD, commit shape) is
language-agnostic. The bug catalog has per-layer stubs with
stack-specific addenda. The issue framework assumes GitHub
but the templates translate to GitLab/Linear/Jira with
minimal edits.

## Use on your own codebase

The framework is most valuable when applied to a real
codebase over weeks — not when read in one pass. Below is
what each primitive is *for* in day-to-day work, and how
they fit together. None of this requires `init.sh`; you
can copy `core/` into an existing repo without touching the
bootstrap scripts.

### `core/complexity.md` — read first

The YAGNI-vs-Ousterhout reconciliation that the framework
otherwise leaves implicit. Read it once at adoption; cite
section numbers in code review when the net-complexity-gain
test applies. Operationally:

- Apply §1 (net-complexity-gain test) before adding any
  abstraction in a code review.
- Cite §3 (pull complexity downward; deep modules) when
  proposing to widen an interface.
- Cite §5 (rule of three) when proposing to extract a
  helper after seeing only one duplication.
- Cite §7 (theory as deliverable) when handing off a
  subsystem: the source is not the artifact; the ADRs and
  glossary entries are.

### `core/session-protocol.md` — at session start

The five rules are loaded by the agent at the start of
every session. The live one for adoption work is Rule 3
("YAGNI for *features*, not for interfaces") paired with
`complexity.md` §1. Refine it for your codebase: are there
particular speculative practices you want banned up front
that the framework's default wording doesn't catch?

### `core/feature-protocol.md` — every feature slice

The deep-module discipline is the operational rule. Pair it
with `core/rpci.md` for any non-trivial work (3+ files, or
design questions). The 3-tier commit rule — Tier 1 (whole
feature), Tier 2 (vertical slice), Tier 3 (apply-site) —
tells the agent when to slice further vs ship whole.

### `core/tdd.md` — every feature or bug fix

Source-anchored red-green-refactor. A failing test pinned to
the user-facing acceptance criterion before the slice
lands. This is what makes AI-assisted commits reviewable.

### `core/adr-discipline.md` — every durable decision

Use the lightweight 4-heading template (Status / Context /
Decision / Consequences) with the next number in sequence.
Promote to a `law` (in `core/laws.md`) when a bug ships
that the ADR would have prevented.

### `skills/consensus-hunter` — pre-merge risk scans

The new parallel-numeric LLM-augmented bug scanner. Pairs
with `bug-hunter` (sequential, narrative, ephemeral) but
is faster and produces persisted per-coord risk scores.

When to run:

- **Pre-merge on a non-trivial branch.** Standalone, this
  is the most valuable operational use of the skill. Run
  it once on the diff; treat the top-10% coords by
  consensus_score as a refresh list for the human reviewer.
- **Pre-release on the whole repo at a known cutoff.**
  Anchors the historical bug-risk ledger that
  `complexity.md` §10 (entropy) prescribes. Each run's
  output lands in `history/<timestamp>.json`; diff two
  consecutive runs to see which coords are rising.
- **Calibration replay.** Run `--calibrate <repo>
  --cutoff YYYY-MM-DD` against a known-buggy period to
  measure Skill Score + precision@K on your real codebase.
  Use this to set per-agent weights in
  `.consensus-hunter/calibration.json` and apply them on
  subsequent runs.

How to run it cheaply: use the `3-agent-min` preset for
pre-commit inner loops, the `5-agent-preset` for pre-merge
audits. Set `--max-concurrent 3` (or whatever your LLM
client allows) — the launcher splits 5-agent presets into
3+2 automatically.

Validation: 33 invariant tests ship with the skill.
Run them with

```bash
python -m unittest discover -t skills/consensus-hunter \
  skills/consensus-hunter/tests
```

Reference calibration: on django @ 2020-06-01, the 5-agent
preset achieved Skill Score +0.2382 and precision@K +0.2222
above base rate on a balanced 36-coord held-out set
(see `.consensus-hunter/REPORT_real_agents.md` if you've
run the calibration harness locally; reproducible from
`scripts/run_real_agent_calibration.py`).

### What this looks like in practice

Once adopted, an LLM agent session on your repo starts by
loading tier-0 docs (`session-protocol.md`, `laws.md`,
`commit-and-branch.md`, `glossary-discipline.md`) plus the
tier-1 doc matching the task. The agent cites principle
numbers in review comments; you cite section numbers in
yours. Decisions that survive a session land in
`docs/adr/`; decisions that survive a release land in
`laws.md`. The framework's value compounds — each session
inherits the previous session's bounded context (via ADRs
and the glossary), so the cost of change falls over time
rather than rising.

For the *complete* design rationale (why YAGNI and
deep-modules don't conflict, why we calibrate to Skill
Score not raw accuracy, why evidence-slices > temperature
for diversity, why logit-space > arithmetic mean for
aggregation), see `core/complexity.md` and the inline
headers in each `lib/` file under `skills/consensus-hunter/`.
